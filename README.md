> 🚀 A structured, hands on JavaScript lab built for **CODOC (PRIVATE) LIMITED's Intern Development Programme**, Assignment 5. It demonstrates **real world AI assisted software development** using prompt engineering techniques across **6 progressive missions**. Every step follows the full professional Git workflow:

```
📌 Issue  ->  🌿 Branch  ->  💻 Code  ->  📝 Commit  ->  🔀 PR  ->  🔍 Review  ->  ✅ Merge  ->  🔒 Close
```

🎯 The goal of this assignment was never to build a large application. The real focus was prompt quality, context management, AI output verification, debugging discipline, and honest GitHub workflow evidence, exactly as required by the assessment brief.

## 🎯 6 Missions Completed

<div align="center">

<img src="https://readme-typing-svg.demolab.com?font=Fira+Code&size=18&pause=800&color=FF6B6B&center=true&vCenter=true&width=500&lines=Click+each+mission+to+expand;All+6+missions+completed" alt="Mission hint" />

<br/>

<details>
<summary><b>🔐 Mission 1: Bad Prompt vs Professional Prompt</b> &nbsp; <img src="https://img.shields.io/badge/Status-Completed-brightgreen?style=flat-square" /></summary>

<br/>

**🌿 Branch:** `docs/bad-vs-good-prompts`

**🎯 Goal:** Compare a bad, better, and professional prompt for generating a password strength checker.

**💡 Key Learning:** A professional prompt using **Role + Task + Constraints + Output Format + Edge Cases** produces a fully safe, reusable, and well documented function.

**📁 Files:**
- `prompts/01-bad-vs-good-prompts.md`
- `src/passwordStrength.js`

**⚡ Prompt Evolution:**

| 🔴 Bad Prompt | 🟡 Better Prompt | 🟢 Professional Prompt |
|:-------------|:----------------|:----------------------|
| "check password" | "check if password is strong" | Role + Task + Constraints + Output Format + Edge Cases |
| Vague output | Basic boolean | Structured object with score, status, feedback |
| No edge cases | Some validation | Full type checking plus empty and null handling |

```javascript
// 🟢 Professional output: Handles types, gives score + status + feedback
checkPasswordStrength("Ab1!defg")
// Returns: { score: 5, status: "Strong", feedback: [], details: { ... } }

checkPasswordStrength(null)
// Returns: { score: 0, status: "Invalid", feedback: ["Input must be a string"] }
```

</details>

<br/>

<details>
<summary><b>✅ Mission 2: Constraint Based Code Generation</b> &nbsp; <img src="https://img.shields.io/badge/Status-Completed-brightgreen?style=flat-square" /></summary>

<br/>

**🌿 Branch:** `feature/form-validator`

**🎯 Goal:** Generate a reusable form validator using explicit constraints. No DOM manipulation, no external libraries, structured error return.

**💡 Key Learning:** Constraints prevent AI from adding unnecessary features, DOM references, or incorrect assumptions.

**📁 Files:**
- `prompts/02-code-generation-prompts.md`
- `src/validator.js`

**🔒 Constraints Applied in Prompt:**

```
✅ Pure JavaScript function (no DOM, no fetch, no imports)
✅ Returns structured { isValid: boolean, errors: object }
✅ Validates: name, email, phone, password, confirmPassword
✅ Multiple error messages per field supported
✅ Strict type checking on all inputs
```

```javascript
// Valid form
validateForm({ name: "John", email: "john@example.com", phone: "03001234567",
               password: "Secure@123", confirmPassword: "Secure@123" })
// Returns: { isValid: true, errors: {} }

// Invalid email
validateForm({ name: "John", email: "not-an-email", ... })
// Returns: { isValid: false, errors: { email: ["Email format is invalid."] } }
```

</details>

<br/>

<details>
<summary><b>🐛 Mission 3: Debugging with AI</b> &nbsp; <img src="https://img.shields.io/badge/Status-Completed-brightgreen?style=flat-square" /></summary>

<br/>

**🌿 Branch:** `fix/discount-calculation`

**🎯 Goal:** Provide a broken discount calculator with full context and expected vs actual results. Ask AI to fix only what is broken.

**💡 Key Learning:** Context rich debug prompts produce more targeted and minimal fixes compared to vague bug reports.

**📁 Files:**
- `prompts/03-debugging-prompts.md`
- `src/discountCalculator.js`

**🔍 Bug Report Structure Used in Prompt:**

```
📋 Context:    calculateDiscount(price, discountPercent, couponCode)
❌ Actual:     calculateDiscount(5, 10, "SAVE10") returns -5.5
✅ Expected:   Should return 0 (clamped, never negative)
🔒 Constraint: Fix only the clamping logic, do not change anything else
```

```javascript
calculateDiscount(100, 10, "SAVE10") // 80   ✅ (10% off then coupon)
calculateDiscount(5, 10, "SAVE10")   // 0    ✅ (clamped, not -5.5!)
calculateDiscount(-1, 10)            // throws Error ✅
calculateDiscount(100, 110)          // throws Error ✅
```

</details>

<br/>

<details>
<summary><b>♻️ Mission 4: Refactoring Prompt</b> &nbsp; <img src="https://img.shields.io/badge/Status-Completed-brightgreen?style=flat-square" /></summary>

<br/>

**🌿 Branch:** `refactor/text-analyzer-cleanup`

**🎯 Goal:** Take a messy, single letter variable named text analyzer and refactor it using an AI prompt specifying clean code standards without changing any output.

**💡 Key Learning:** Specifying **"no new features, same return schema"** prevents scope creep and keeps refactoring focused.

**📁 Files:**
- `prompts/04-refactoring-prompts.md`
- `src/textAnalyzer.js`

**📊 Refactoring Results:**

| 🔴 Before | 🟢 After |
|:---------|:--------|
| Single letter variable names (`w`, `c`) | Descriptive names (`wordCount`, `charCount`) |
| No comments or docs | JSDoc on every function |
| Deeply nested logic | Extracted helper functions |
| No null/undefined guards | Full edge case handling |

```javascript
analyzeText("hello world hello")
// Returns: { wordCount: 3, charCount: 17, charCountNoSpaces: 15, mostFrequentWord: "hello" }
```

</details>

## 💡 Prompting Techniques

<div align="center">
<img src="https://readme-typing-svg.demolab.com?font=Fira+Code&size=16&pause=700&color=FF6B6B&center=true&vCenter=true&width=500&lines=7+Techniques+Mastered;Applied+Across+6+Missions" alt="Techniques banner" />
</div>

<br/>

| 🧠 Technique | 🎯 Missions Used | 📋 Description |
|:------------|:----------------|:--------------|
| 🎭 **Role Assignment** | All (M1 to M6) | Assign AI a specific persona (Senior Mentor, Reviewer, Security Lead) |
| 🔒 **Constraint Injection** | M2, M4, M5 | Explicit do and don't rules to bound AI output scope |
| 🔍 **Context + Expected/Actual** | M3 | Provide full context with before/after expected behavior |
| 📐 **Output Format Specification** | All (M1 to M6) | Define exact return shape (object structure, keys, types) |
| 🧪 **Edge Case Enumeration** | M1, M2, M3 | List all edge cases AI must handle explicitly |
| 🩹 **Minimal Fix Instruction** | M3 | Tell AI exactly what NOT to change to avoid over engineering |
| 📜 **Policy Drafting** | M6 | Use AI to structure compliance documents with human final review |

## 🎓 Learning Summary

> 🧠 The biggest lesson from this assignment was the gap between **AI assisted** and **AI dependent** work. AI can generate a first draft fast, but every function still needed manual testing, edge case checking, and a clear explanation before it could be trusted.

**Key takeaways:**

- 🎯 A well structured prompt (Role, Task, Context, Constraints, Output Format, Edge Cases, Verification) consistently beats a vague one, even on the first try
- 🔍 Manual testing caught issues AI missed on its own, especially around clamping, null handling, and type checks
- 🩹 Telling AI exactly what **not** to change kept fixes minimal and prevented scope creep during debugging and refactoring
- 📜 AI is useful for drafting policy and documentation, but human review is still required for accuracy and accountability
- 🔢 Reviewing prompts before sending them, instead of regenerating repeatedly, kept token usage efficient without sacrificing quality
